# 基于嵌入式 Linux 的智能无人零售终端系统完整技术路线

> 项目定位：面向嵌入式 Linux 应用层岗位的综合型项目。  
> 推荐项目名：**基于 IMX6ULL 的智能无人零售终端系统** / **Smart Retail Terminal Based on Embedded Linux**。

---

## 1. 项目背景与定位

传统无人零售设备通常包含本地交互、商品管理、订单生成、支付状态、出货控制、设备监控、异常告警、远程管理和版本升级等功能。

本项目不建议做复杂的“AI 无人超市”，也就是不做多摄像头行为识别、商品自动识别和真实支付闭环，而是做一个更适合嵌入式 Linux 应用层求职的工程项目：

```text
用户在终端屏幕选择商品
        ↓
终端生成订单并模拟支付
        ↓
IMX6ULL 控制出货模块
        ↓
V4L2 摄像头抓拍订单图片
        ↓
SQLite 本地保存订单和库存
        ↓
MQTT 上报设备状态和订单事件
        ↓
TCP/HTTP 上传抓拍图片
        ↓
Qt 后台管理商品、订单、库存、设备状态和 OTA 升级
```

项目核心不是“无人超市”这个概念，而是通过一个真实业务场景，把嵌入式 Linux 应用层常用技术串起来：

- 多线程
- TCP 网络通信
- MQTT 物联网通信
- JSON/cJSON 协议
- SQLite 本地持久化
- V4L2 摄像头采集
- LVGL 本地 UI
- GPIO/继电器/蜂鸣器控制
- Qt 后台管理端
- OTA 远程升级
- 状态机
- 日志系统
- 断网缓存与补传

---

## 2. 项目总体目标

实现一套运行在 IMX6ULL 开发板上的智能无人零售终端系统，具备以下能力：

1. 本地商品展示和购买交互。
2. 模拟支付成功后生成订单。
3. 根据商品货道控制 GPIO/继电器/LED/蜂鸣器完成出货模拟。
4. 使用 V4L2 摄像头完成订单抓拍。
5. 使用 SQLite 保存商品、库存、订单和日志。
6. 支持断网订单缓存和网络恢复后自动补传。
7. 使用 MQTT 完成设备心跳、状态上报、订单上报、库存同步和远程控制。
8. 使用 TCP/HTTP 完成订单图片上传。
9. 使用 Qt 后台完成设备监控、订单管理、商品管理、库存管理、日志查看和 OTA 推送。
10. 支持应用级 OTA 升级，包括版本检测、升级包下载、SHA256 校验、安装替换和失败回滚。

## 5. 核心业务流程

### 5.1 用户购买流程

```text
用户进入商品页面
        ↓
点击某个商品
        ↓
系统检查库存
        ↓
库存充足，生成订单
        ↓
模拟支付成功
        ↓
订单状态变为 PAID
        ↓
控制出货模块
        ↓
出货成功后触发摄像头抓拍
        ↓
保存订单图片
        ↓
SQLite 保存订单记录
        ↓
MQTT 上报订单事件
        ↓
TCP/HTTP 上传图片
        ↓
后台显示订单和抓拍图片
        ↓
库存减少，UI 刷新
```

### 5.2 断网缓存流程

```text
订单生成
        ↓
检测网络是否在线
        ↓
在线：立即上报 MQTT，并上传图片
        ↓
离线：订单写入 SQLite，upload_status = pending
        ↓
网络恢复
        ↓
查询 pending 订单
        ↓
自动补传订单和图片
        ↓
补传成功后更新 upload_status = uploaded
```

### 5.3 后台远程控制流程

```text
Qt 后台点击远程开货道
        ↓
后台发布 MQTT 控制消息
        ↓
终端 mqtt_thread 收到消息
        ↓
解析 JSON
        ↓
投递到 device_thread
        ↓
GPIO/继电器动作
        ↓
终端发布 control_ack
        ↓
Qt 后台显示执行结果
```

### 5.4 OTA 升级流程

```text
Qt 后台点击推送 OTA
        ↓
后台发布 ota_notify MQTT 消息
        ↓
终端收到 OTA 通知
        ↓
下载 version.json
        ↓
比较当前版本和最新版本
        ↓
HTTP 下载 tar.gz 升级包
        ↓
SHA256 校验
        ↓
备份旧程序
        ↓
解压并替换新程序
        ↓
重启应用
        ↓
启动成功，写入 app_ok
        ↓
上报 OTA success

如果启动失败：
        ↓
守护脚本检测不到 app_ok
        ↓
恢复 backup 旧程序
        ↓
重启旧版本
        ↓
上报 rollback
```

## 6. 状态机设计

### 6.1 设备状态机

```c
typedef enum {
    DEVICE_IDLE = 0,
    DEVICE_SELECTING,
    DEVICE_PAYING,
    DEVICE_DISPENSING,
    DEVICE_CAPTURING,
    DEVICE_UPLOADING,
    DEVICE_SUCCESS,
    DEVICE_ERROR
} device_state_t;
```

状态说明：

| 状态 | 含义 |
|---|---|
| DEVICE_IDLE | 空闲状态 |
| DEVICE_SELECTING | 用户选择商品 |
| DEVICE_PAYING | 模拟支付中 |
| DEVICE_DISPENSING | 出货中 |
| DEVICE_CAPTURING | 摄像头抓拍中 |
| DEVICE_UPLOADING | 订单或图片上传中 |
| DEVICE_SUCCESS | 购买完成 |
| DEVICE_ERROR | 设备故障 |

### 6.2 订单状态机

```c
typedef enum {
    ORDER_CREATED = 0,
    ORDER_PAID,
    ORDER_DISPENSING,
    ORDER_CAPTURED,
    ORDER_UPLOADING,
    ORDER_DONE,
    ORDER_FAILED,
    ORDER_PENDING_UPLOAD
} order_state_t;
```

订单状态流转：

```text
CREATED
  ↓
PAID
  ↓
DISPENSING
  ↓
CAPTURED
  ↓
UPLOADING
  ↓
DONE
```

异常情况：

```text
任意状态失败 → ORDER_FAILED
网络失败 → ORDER_PENDING_UPLOAD
```

typedef enum {

## 7. 线程模型设计

### 7.1 推荐线程架构

```text
main
 ├── ui_thread
 ├── order_thread
 ├── device_thread
 ├── camera_thread
 ├── mqtt_thread
 ├── tcp_upload_thread
 ├── ota_thread
 ├── storage_thread
 └── log_thread
```

### 7.2 各线程职责

| 线程 | 职责 |
|---|---|
| ui_thread | LVGL 刷新、处理本地触摸事件、显示商品/订单/状态 |
| order_thread | 订单创建、状态流转、库存扣减、业务流程控制 |
| device_thread | GPIO、继电器、蜂鸣器、LED、出货动作 |
| camera_thread | V4L2 摄像头抓拍、图片保存 |
| mqtt_thread | MQTT 连接、订阅、发布、断线重连 |
| tcp_upload_thread | 图片上传、订单补传、大文件传输 |
| ota_thread | OTA 检查、下载、校验、安装、回滚 |
| storage_thread | SQLite 读写、缓存订单、查询 pending 数据 |
| log_thread | 异步日志落盘、日志轮转 |

### 7.3 模块间通信方式

推荐使用安全队列传递事件：

```text
UI 点击购买
    ↓
投递 ORDER_CREATE_EVENT 到 order_queue
    ↓
order_thread 处理订单
    ↓
投递 DEVICE_DISPENSE_EVENT 到 device_queue
    ↓
device_thread 控制出货
    ↓
投递 CAMERA_CAPTURE_EVENT 到 camera_queue
    ↓
camera_thread 抓拍
    ↓
投递 NETWORK_UPLOAD_EVENT 到 upload_queue
```

推荐公共事件结构：

```c
typedef enum {
    EVENT_ORDER_CREATE = 0,
    EVENT_ORDER_PAID,
    EVENT_DEVICE_DISPENSE,
    EVENT_CAMERA_CAPTURE,
    EVENT_MQTT_REPORT,
    EVENT_UPLOAD_IMAGE,
    EVENT_OTA_START,
    EVENT_SYSTEM_ERROR
} event_type_t;

#define EVENT_DATA_SIZE 256

typedef struct {
    event_type_t type;
    char order_id[64];
    char data[EVENT_DATA_SIZE];
} app_event_t;
```

## 8. MQTT 设计

### 8.1 MQTT 在项目中的作用

MQTT 用于物联网设备通信，主要负责：

- 设备心跳
- 设备状态上报
- 订单事件上报
- 库存状态上报
- 异常告警
- 后台远程控制
- OTA 升级通知
- OTA 进度上报

不建议用 MQTT 传输大图片和 OTA 文件。

推荐分工：

| 数据类型 | 通信方式 |
|---|---|
| 心跳 | MQTT |
| 状态 | MQTT |
| 订单事件 | MQTT |
| 库存同步 | MQTT |
| 远程控制 | MQTT |
| OTA 通知 | MQTT |
| OTA 进度 | MQTT |
| 图片上传 | TCP / HTTP |
| OTA 包下载 | HTTP |

## 9. OTA 设计

### 9.1 OTA 类型选择

本项目建议做 **应用级 OTA**，不做完整系统固件 OTA。

升级对象：

```text
/opt/smart_retail/app/SmartRetailTerminal
```

不升级内容：

```text
Linux Kernel
RootFS
U-Boot
设备树
```

原因：

1. 应用级 OTA 更容易实现。
2. 更适合嵌入式 Linux 应用层岗位展示。
3. 风险低，不容易把板子刷坏。
4. 可以体现版本管理、下载、校验、替换、回滚等企业级能力。

### 9.2 OTA 服务器文件结构

```text
ota_server/
├── version.json
├── SmartRetailTerminal_v1.0.1.tar.gz
└── SmartRetailTerminal_v1.0.2.tar.gz
```

## 13. 模块职责划分

### 13.1 app 模块

负责系统初始化、线程创建、全局上下文管理和系统状态流转。

- 初始化日志
- 初始化数据库
- 初始化网络模块
- 初始化 UI 模块

### 13.2 ui 模块

负责 LVGL 本地交互界面。

页面建议：

1. 首页：显示设备状态、网络状态、当前时间。

### 13.3 order 模块

负责订单业务逻辑。

主要功能：

- 创建订单
- 检查库存
- 模拟支付
- 更新订单状态
- 触发出货
- 触发抓拍

### 13.4 network 模块

负责 TCP/HTTP 通信。

- HTTP 上传图片

### 13.5 mqtt 模块

负责 MQTT 物联网通信。

主要功能：

- 连接 Broker

## 14. 开发阶段规划

### 阶段 1：PC 端业务模拟

目标：先不依赖板子，跑通核心业务。

内容：

1. 商品结构体。
2. 订单结构体。
3. 订单状态机。
4. SQLite 保存订单。
5. TCP 发送订单到 Qt 后台。
6. Qt 后台显示订单。

完成标准：

```text
终端程序运行后，可以生成订单，并在 Qt 后台看到订单记录。
```

### 阶段 2：LVGL 本地界面

目标：让用户可以通过屏幕购买商品。

内容：

1. LVGL 首页。
2. 商品列表页。
3. 购买按钮。
4. 订单状态显示。
5. 设备状态显示。

完成标准：

```text
点击 LVGL 商品按钮后，可以触发订单流程。
```

### 阶段 3：V4L2 摄像头抓拍

目标：每个订单绑定一张抓拍图片。

内容：

1. camera_if 抽象接口。
2. camera_mock 模拟实现。
3. camera_v4l2 实际实现。
4. 购买成功后抓拍 JPG。
5. 图片路径写入订单表。

完成标准：

```text
每次购买成功后，data/snapshots/ 下生成一张订单图片。
```

### 阶段 4：设备控制

目标：用 GPIO/继电器/LED 模拟货道出货。

内容：

1. gpio_ctrl。
2. relay_ctrl。
3. buzzer_ctrl。
4. dispenser 货道控制。
5. 不同商品对应不同 slot_id。

完成标准：

```text
购买不同商品时，不同 GPIO 或 LED 动作。
```

### 阶段 5：MQTT 物联网通信

目标：让终端具备设备接入能力。

内容：

1. Mosquitto Broker 搭建。
2. mqtt_client 初始化。
3. 心跳上报。
4. 订单上报。
5. 状态上报。
6. 后台远程控制。
7. 库存同步。

完成标准：

```text
Qt 后台通过 MQTT 能看到设备在线，并能下发远程开货道命令。
```

### 阶段 6：断网缓存与补传

目标：模拟真实设备网络异常。

内容：

1. 网络状态检测。
2. 离线订单缓存。
3. pending 订单查询。
4. 网络恢复后补传。
5. 补传成功更新状态。

完成标准：

```text
断网购买不会丢订单，网络恢复后订单自动上传。
```

### 阶段 7：OTA 远程升级

目标：实现应用级 OTA。

内容：

1. version.json。
2. MQTT OTA 通知。
3. HTTP 下载升级包。
4. SHA256 校验。
5. 备份旧程序。
6. 替换新程序。
7. systemd 重启。
8. 启动失败回滚。
9. OTA 状态上报。

完成标准：

```text
Qt 后台点击升级后，终端下载新版本程序并完成替换；如果升级失败，可以回滚旧版本。
```

### 阶段 8：项目打磨与演示

目标：让项目像企业项目。

内容：

1. 完善日志。
2. 完善异常码。
3. 完善 README。
4. 绘制架构图。
5. 录制演示视频。
6. 整理简历描述。

```text

5. 基于 V4L2 实现订单抓拍功能，并通过 TCP/HTTP 上传图片到后台管理端。
6. 基于 LVGL 实现嵌入式终端交互界面，支持商品选择、购买状态和设备状态显示。
7. 基于 Qt 实现后台管理系统，支持设备监控、订单查看、库存管理、MQTT 消息监控和 OTA 升级推送。
```

它可以作为嵌入式 Linux 应用层方向的核心求职项目。

项目更准确的名字不建议叫“无人超市”，而建议叫：

```text
基于嵌入式 Linux 的智能无人零售终端系统
```

或者：

```text
基于 IMX6ULL 的智能零售柜终端系统
```

这样更真实、更好讲，也更符合嵌入式 Linux 应用层岗位的能力要求。
